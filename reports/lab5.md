## Цель работы

Познакомиться с простыми техническими средствами управления проектами на
основе задач (tickets/issues).

## Этапы выполнения работы

После наведения курсором и нажатия на дефолтное имя проекта были внесены изменения, имя стало `Kursovaya rabota tic-tac-toe`. Во вкладке `Settings -> Manage access` преподаватель был приглашен коллаборатором в роли "write" (может комментировать, делать ревью, двигать задачи,работать с PR).

2. Во вкладке `View (шестеренка)` был выбран вид "доски" для проекта. Показались три колонки с названиями: `To do`, `In progress`, `Done`. Вместо них, с помощью кнопки `+`, были созданы следующие колонки:

  - Запланировано к выполнению;
  - В процессе;
  - На проверке;
  - Нужно исправить;

3. Были добавлены поля в задачи, в которых можно отмечать предполагаемые
   даты начала и окончания, с помощью `View -> Fields -> New field`. Далее, выбор типа поля -- дата. 

4. Было добавлено поле в задачи, которые отражают тип работы:
   исследовательская работа, составление документации, программирование. С помощью `View -> Fields -> New field -> Single select`. 

### Планирование работ по курсовой работе

1. Была декомпозирована работа над курсовым проектом по "Структурам данных" на
   небольшие задачи, каждую из которых можно сделать за 4-10 часов. Каждая задача имеет
   короткое описание, в котором сформулировано, что требуется и какие
   артефакты должны получиться по итогу выполнения.

2. Были созданы тикеты в терминах проекта Github для декомпозированных задач,
   заданы нужные типы и проставлены примерные сроки. Все тикеты на данный момент расположены в колонке "Существующие задачи". Чтобы создать тикет необходимо: `В нужной колонке "+ Add item" -> Enter -> Blank issue -> Ввести название и описание -> Create`. Чтобы проставить настроенные ранее поля, необходимо нажать на интересующую задачу, в появившемся окне расставить даты, выбрать тип работы. Таким образом были созданы N задач, в каждой поставлены даты, что будет итогом, а также, по мере необходимости, зависимости от других задач (создавались с помощью конструкции `- [ ] #29 `, где `-` пункт безымянного списка, `[ ]` место для галочки выполнено, `# N` номер задачи и ссылка на нее, созданная гитхабом):
   - Анализ постановки задачи курсовой работы
   - Анализ структуры исходного кода
   - Исследование возможных алгоритмов
   - Разработка алгоритма поведения бота
   - Проверка алгоритма вручную
   - Подготовка алгоритма + согласование
   - Реализация структуры бота
   - Реализация хоть-какого-нибудь хода бота
   - Реализация основного алгоритма бота
   - Проверка корректности работы бота
   - Оценка качества игры бота
   - Защита реализации бота

   - Подготовка отчета к беловому варианту
   - Защитить отчет у преподавателя

### Выполнение задачи

1. Для задачи, которая а) имеет результатом текст (например,
   постановку, алгоритм или код), б) требует согласования необходимо создать черновой
   запрос на слияние, привязанный к этой задаче. Чтобы создать черновой PR, необходимо при создании обычного PR `нажать на стрелочку возле Create pull request -> Create draft pull request`. Draft PR нельзя случайно объединить с основной веткой, пока он находится в статусе черновика. В описании PR необходимо оставить `спец.слово` и ссылку на задачу `# номер`. Тогда при мёрдже этой ветке в основную, задача автоматически закроется. Спец слова (глобально они имеют единый смысл - после успешного слияния с итоговой веткой PR закрывают задачу автоматически, однако оттенки смыслов разные, это удобно): 
   - Fixes — для багов (ошибок);
   - Closes — для обычных задач или обновлений документации;
   - Resolves — для сложных вопросов или дискуссий.

2. Необходимо добавить преподавателя в запрос в качестве ревьювера.

3. Тикет задачи необходимо по доске соответственно состоянию вашей задачи в
   процессе, например, в работе -- переместить в соответствущую колонку.

4. Когда задача на согласовании, необходимо перейти на страницу с draft PR и нажать на кнопку `Ready for review`, задача готова к ревью. 

5. По итогам ревью необходимо перетаскивать задачу в нужную стадию на доске, до тех пор пока запрос на
   слияние не будет принят. После этого благодаря конструкции `спец.слово #номерзадачи`, указанной в описании PR, задача автоматически закроется.

### Исследование альтернатив

1. Было выяснено, какие еще решения для управления проекта можно использовать для
   реализации похожего процесса.

   - Jira — это ведущий инструмент для управления проектами и отслеживания задач. Включает готовые отчеты, в систему интегрирован искусственный интеллект, реализует методологию Agile. С 2022 года компания Atlassian (отв. за Jira) прекратила продажу новых лицензий в России, то есть пользователи продолжают работать без официальной поддержки и обновлений. 
   - Kaiten — это российская платформа для управления проектами и визуализации рабочих процессов. Сервис объединяет функции трекера задач, базы знаний, позволяет командам организовывать работу в едином цифровом пространстве. Платформа ориентирована на гибкие подходы Agile. У сервиса есть бесплатная версия (Free) с базовым набором функций для небольших команд, а также платные корпоративные тарифы с расширенной аналитикой, безопасностью и дополнительными модулями. 
   - Trello — это популярный облачный сервис для управления проектами и личными задачами, основанный на методологии канбан-досок. Система строится на трех основных элементах: доски, списки, карточки. Компания Atlassian (владелец Trello) прекратила работу в России и Беларуси. Регистрация новых аккаунтов и оплата подписок российскими картами напрямую недоступны. На бесплатном тарифе Trello есть всё необходимое для личного планирования или работы небольшой команды, но с ограничениями по количеству досок и автоматизации.

2. В качестве альтернативы системе управления проектами, встроенной в GitHub, была рассмотрена платформа Trello, основанная на досках.

    Trello представляет собой инструмент визуального управления задачами, в котором работа организуется в виде досок, списков и карточек. Каждая карточка соответствует отдельной задаче и может содержать описание, сроки выполнения, вложения, комментарии и участников.

    Необходимо создать новую доску. На доске формируются списки, соответствующие этапам выполнения задач:

    - список всех задач
    - запланированные задачи
    - задачи в работе
    - задачи на проверке
    - задачи на доработке
    - завершённые задачи

    Каждая задача переносится в Trello в виде отдельной карточки. В карточке указывается:

    - название задачи
    - описание 
    - тип задачи может быть реализован с помощью меток (labels)
    - предполагаемые сроки выполнения с помощью поля Due Date

    Зависимости между задачами можно указать в описании карточек, например, с помощью ссылок или текстовых пометок.

    В процессе выполнения задачи карточка перемещается между списками.

    Для согласования результатов преподаватель может быть добавлен в карточку как участник и оставлять комментарии.

    Trello позволяет отслеживать статус задач в реальном времени, а также организовывать обсуждение внутри карточек.

    Бесплатная версия Trello имеет ограничения:
    - ограниченное количество расширений (буквально пару штук)
    - отсутствие встроенных средств для работы с кодом 

    Trello ориентирован исключительно на управление задачами, тогда как GitHub дополнительно предоставляет инструменты для разработки и контроля версий кода.
